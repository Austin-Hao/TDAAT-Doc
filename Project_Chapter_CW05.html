1. Objective

1.1 Management perspective

OBJ-MAN

How this project will be judged as successful?

  • OBJ-MAN-1: Improve time & cost efficiency of In-car road testing.
  • OBJ-MAN-2: Increase test evaluation accuracy during In-car road test process.
  • OBJ-MAN-3: To standardize In-car road test process & protocol.
  • OBJ-MAN-4: To accumulate lesson learnt/experience for future testing projects.
  • OBJ-MAN-5: To be a part of PMT to support testing methodology.

OBJ-LAH

Project aim described in LAH:

  • OBJ-LAH-1: Test report could be created automatically.

  • OBJ-LAH-3: Tester could import the test data for analysis automatically.

2.2 Sections and WBS

Due to ensure the overall system design and arrangement of development work, the 6 packages are broken down into following 3 sections.

 

Execution Section:

This is the core section of the project, it focus on all functions during the on-car test:

For instance:

  • Overview of the test completion
  • Vehicle status and signal monitoring
  • Test execution instructions
  • Trace and video recording etc.

 

Test Script Section:

This part focus on helping tester transfer nature language described test case into test script.

For instance:

  • Import standard formated test case file
  • Test script generation

 

Management Section:

This part focus on management-related functions, and the interaction with codeBeamer.

For instance:

  • Test script management
  • Test set management
  • Test results management
  • Report generation
  • Test data analysis

 

Those 3 sections does NOT means 3 different APPs, it well be decided in the later phase that which part should be integrated.

 

 

 

3. Project Approach & Milestone

DEV STAGE 1 CURRENT

Focusing on technicial preparation, UI design, file format and basic software frame.

Inputs for the develepment should be well collected, defined and clearified in this stage.

At the end of this stage, following checkpoints should be reached:

  • Test Script Format

    • Include the standard test case format, which will be used as input of script generation.
    • Fields and structure of test set and test run.

  • Signal Monitor Framework

    • Basic data processing according to the Kmatrix.
    • Draft UI design.
    • Work flow, which should be agreed by the users.

  • Basic Management

    • Basic management framework
    • Draft UI design.
    • Work flow, which should be agreed by the users.

 

DEV STAGE 2

(May will be adjusted at the end of stage 1)

The core functions of the testing tool should be built in this stage, field test should be organized, and after this stage, all crews should be confident that we are on the right way to the objectives without any significant risk.

At the end of this stage, following checkpoints should be reached:

  • Generate Test Script

    • With the formated test case input, the test script should be generated.
    • It may have some special cases which need to be handled.

  • Run Sample Test Script on MLB

    • Combined software with hardware on car, the execution system should be able to run single test script.
    • Any unexpected challenges in early stage should be exposed through the on-car field test.

  • Version, Traceability, Result and Report

    • All main management functions should be able to demostrated.
    • Review test results
    • Generate test report

 

DEV STAGE 3

(May will be adjusted at the end of stage 2)

Finish the development of all functions, the integration and release preparation is also a main topic in this stage, related document should be prepared.

At the end of this stage, following checkpoints should be reached:

  • Compose Test Script

    • A friendly UI to write stantard formated test case, with cloud be converted to script and also uplaod to codeBeamer.

  • Implemetation to other carline

    • The system could support other carline

  • Data Server

    • A data server should be built to enable basic data analysis of test data.

 

BUG FIX

After the official release, the development team will focus on bug fixing. Bug 0 should be reached at the end of this stage.

b. Requirements in LAH:

REQT-LAH-A

A: Test case edit toolkit

OBJ-MAN-1 / OBJ-MAN-3 / OBJ-MAN-5 / OBJ-LAH-3

  • A-10: Test case could be managed in this toolkit.
  • A-20: This toolkit could help the test manager to transfer the nature language described test cases automatically into signal language described test case (Test case ->Test script).
  • A-30: In order to make this transfer more accuracy and efficiency, the nature language described test cases could follow defined grammatical rules or graphic guidance to generate the test scripts, which should be easy and simple to follow.
  • A-40: This toolkit provide the opportunity to manage the changes and versions of test cases.
  • A-50: This toolkit could manage and monitor the traceability between test cases and function require- ments.
  • A-60: Test script could be generated, edited, deleted, added and synchronized in this toolkit.
  • A-70: This toolkit provide interface to test management tool (codeBeamer), which means the test cases could be transferred between codeBeamaer and this toolkit.

 

REQT-LAH-B

B: Test Data Analysis Automation toolkit

OBJ-MAN-1 / OBJ-MAN-2 / OBJ-MAN-3 / OBJ-MAN-5 / OBJ-LAH-4

  • B-10: During test execution in the car, this toolkit could check & monitor.

    1. test case pre-condition, action and expected reaction.
    2. availability of relevant signals (include Flexray, CAN, FD CAN, LIN, vehicle Ethernet)
    3. status of test case execution (PASS/FAIL/etc.)
    4. status of test case accomplishment.

  • B-20: Toolkit could make error injection through OBD-CAN (for example) bus online.

  • B-30: After test execution finished in car, the test results could be centralized saved and trans- ferred to test codeBeamer.

 

REQT-LAH-C

C: Test Data Auto Logging Toolkit

OBJ-MAN-2 / OBJ-LAH-4

  • C-10: The test data will be saved in a way that the traceability between test cases and correspond- ing test data could be managed.

 

REQT-LAH-D

D: Test Data Processing Toolkit

OBJ-MAN-1 / OBJ-MAN-2 / OBJ-LAH-4

  • D-10: The test data of executed test cases could be processed or post-processed.
  • D-20: This toolkit could pre-check the condition of data processing environment and indicate the test condition.

 

REQT-LAH-E

E: Test report interface toolkit

OBJ-MAN-1 / OBJ-MAN-2 / OBJ-MAN-5 / OBJ-LAH-1 / OBJ-LAH-2

  • E-10: The test report could generated by toolkit for test execution results with detailed signal infor- mation.
  • E-20: The test report could be exported to test management tool codeBeamer.

  • F-50: The GUI could display the following information:

    • Case status: display the status of the test case, divided into:
